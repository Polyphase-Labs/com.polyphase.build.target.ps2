# PS2 Build System ‚Äî Polyphase Build Target

`Makefile_PS2` is the PS2SDK build wrapper. The editor's
`ComPolyphaseBuildTargetPs2` addon invokes it as:

```
make -C <projectDir>/Intermediate/PS2 \
     -f <addonRoot>/Makefile_PS2 \
     PROJECT_ROOT=<projectDir> \
     POLYPHASE_PATH=<engineDir> \
     -j<N>
```

This page documents how the Makefile resolves paths, discovers sources,
embeds `audsrv.irx`, and links the final ELF. Read this if a build fails
in a way the editor's error message doesn't explain.

## Host toolchain ‚Äî PS2DEV

The addon expects PS2DEV in one of these places (resolved in order):

1. `PS2DEV` set on the `make` command line.
2. `PS2DEV` exported in the environment.
3. Common install paths probed by the Makefile:
   - `/usr/local/ps2dev`
   - `/opt/ps2dev`
   - `$HOME/ps2dev`
   - `/mnt/c/ps2dev` (WSL view of Windows C: drive)
   - `/mnt/d/ps2dev`
4. Last-resort default: `/usr/local/ps2dev`.

The Makefile validates by probing for
`$(PS2DEV)/ps2sdk/samples/Makefile.eeglobal_cpp`. If that sentinel is
missing, the build aborts with a message including a download link to the
ps2dev release page. **Do not try to "fix" this with a half-installed
toolchain** ‚Äî ps2dev expects its full directory layout to be intact.

`PATH` is forcibly exported with PS2DEV's binary directories prepended so
recursive `ee-gcc` / `ee-ld` invocations find the toolchain even when the
parent shell didn't set it up. This matters especially under WSL invocations
from the editor.

## Two paths into the toolchain

| Path | When used |
| --- | --- |
| **Native** (Linux, macOS, **Windows**) | Default. ps2dev ships Windows binaries since v2.0.0; the Windows install layout is the same as POSIX. |
| **WSL fallback** (Windows opt-in) | Selected via the **Use WSL** Target Option. The editor pre-translates `C:\path` ‚Üí `/mnt/c/path` so the bash body never sees backslashes. |

Whichever path is taken, the Makefile itself is identical ‚Äî only the
invocation differs.

## Project root + engine path resolution

```
PROJECT_ROOT := (1) command line override
             || (2) walk up from $(CURDIR) looking for a *.octp file (2 levels)
             || error
```

`*.octp` is the Polyphase project marker file. The walk-up handles three
common invocation contexts:

| `$(CURDIR)` | Outcome |
| --- | --- |
| `<project>/` | `$(CURDIR)` has `*.octp` |
| `<project>/Intermediate/PS2/` (default) | Walk up two levels |
| Anywhere else | Aborts with actionable message |

```
POLYPHASE_PATH := (1) command line override
               || (2) environment variable
               || (3) walk up from $(PROJECT_ROOT) looking for Polyphase.sln (6 levels)
               || (4) /opt/Polyphase
```

Both paths are validated by probing for canonical files (`*.octp` and
`Engine/Source/Engine/Engine.h` respectively). A mis-resolved walk-up
that finds a non-project directory triggers an error rather than silently
linking the wrong codebase.

Trailing slashes are stripped via `patsubst %/,%,$(VAR)`. This matters
because `$(notdir foo/)` returns empty in GNU make, which previously
caused the TARGET (`$(notdir $(PROJECT_ROOT))`) to be empty and link
output to a file literally named `.elf`. The `override` keyword is
required because GNU make silently ignores plain `:=` reassignment of
command-line vars.

## Source discovery

PS2SDK's `Makefile.eeglobal_cpp` does NOT auto-wildcard source lists ‚Äî
unlike PSPSDK. The Makefile enumerates files by walking a fixed list of
source directories:

```make
SOURCES = \
    $(POLYPHASE_PATH)/Engine/Source/Engine \
    $(POLYPHASE_PATH)/Engine/Source/Engine/Assets \
    ...
    $(POLYPHASE_PATH)/External/Lua \
    $(POLYPHASE_PATH)/External/Vorbis \
    $(POLYPHASE_PATH)/External/Bullet/... \
    $(POLYPHASE_PATH)/External/recastnavigation/... \
    $(ADDON_PS2_RUNTIME) \
    $(ADDON_PS2_RUNTIME)/Graphics_PS2GS \
    $(PROJECT_ROOT)/Source \
    $(ADDON_SOURCES)

CPPFILES := $(foreach dir,$(SOURCES),$(notdir $(wildcard $(dir)/*.cpp)))
CFILES   := $(foreach dir,$(SOURCES),$(notdir $(wildcard $(dir)/*.c)))
```

Each .cpp/.c becomes a flat `*.o` in `CWD` (= `$(PROJECT_ROOT)/Intermediate/PS2`).
`VPATH` lists the source directories so `make` knows where to find each
source file by basename.

### `EmbeddedScripts.cpp` (yes), `EmbeddedAssets.cpp` (no)

The Makefile conditionally adds `EmbeddedScripts.cpp` (generated by the
editor ‚Äî contains all Lua scripts as `EmbeddedFile` entries):

```make
ifneq ($(wildcard $(PROJECT_ROOT)/Generated/EmbeddedScripts.cpp),)
CPPFILES += EmbeddedScripts.cpp
endif
```

But it **deliberately does NOT add** `EmbeddedAssets.cpp`. The EE has
only 32 MB main RAM, the same budget as the PSP. A cooked-asset blob
(textures, audio, meshes baked into the ELF) would consume more than
half of that before the game even runs. Assets load from host:/cdrom0:/
mc0:/ at runtime via `AssetManager::Discover` instead. See the
`project_psp_scripts_embedded_assets_disk` decision recorded in the
shared memory.

### Engine-target addon injection

```make
-include $(PROJECT_ROOT)/Generated/AddonInject.mk
```

The editor writes `AddonInject.mk` before invoking `GetCompileCommand`.
It defines:

| Variable | Purpose |
| --- | --- |
| `ADDON_SOURCES` | Extra source directories (per addon) |
| `ADDON_DEFINES` | Extra `-D...` defines |
| `ADDON_INCDIR` | Extra `-I...` include paths |
| `ADDON_LIBDIR` | Extra `-L...` link paths |
| `ADDON_LIBS` | Extra `-l...` libs to link |
| `ADDON_CPPFILES_EXTRA` | Extra individual .cpp files (e.g. `AddonPlugins.cpp` registrar) |

The leading `-` on `-include` makes the include non-fatal ‚Äî projects
without addons still build.

## `audsrv.irx` embedding

audsrv is loaded onto the IOP at boot (`Main_PS2.cpp::main`). The IRX
binary itself doesn't live on the BIOS ROM, so it has to ship as part
of the ELF or alongside it. The Makefile embeds it by transforming the
IRX bytes into a C source file via PS2SDK's `bin2c`:

```make
AUDSRV_IRX_SRC := $(PS2SDK)/iop/irx/audsrv.irx

audsrv_irx.c: $(AUDSRV_IRX_SRC) $(ADDON_PS2_RUNTIME)/../../Makefile_PS2
	rm -f $@
	$(PS2SDK)/bin/bin2c $< $@ audsrv_irx
audsrv_irx.o: audsrv_irx.c
	$(EE_CC) $(EE_CFLAGS) -c $< -o $@
EE_OBJS += audsrv_irx.o
```

The dependency on `Makefile_PS2` itself forces a rebuild of the .o when
the recipe changes (GNU make watches file deps by default, not recipe
changes). `bin2c` generates:

```c
unsigned char audsrv_irx[]      = { 0x7f, 0x45, 0x4c, 0x46, ... };
unsigned int  size_audsrv_irx   = sizeof(audsrv_irx);
```

The compiler places the array in `.rodata` (read-only data segment) under
`-O2`, which is what we want ‚Äî the IRX bytes live in the ROM image rather
than being copied to RAM at startup. `Main_PS2.cpp` declares the symbols
`extern "C"` and passes them to `SifExecModuleBuffer`.

### Why not `ld -r -b binary` or `.incbin`?

Both were tried earlier. `ld -r -b binary` produced an .o that the IOP
loader rejected with `-201 (KE_LINKERR)` ‚Äî likely missing alignment or
section-attribute mismatch. `.incbin` works in theory but its path
quoting through successive make/shell layers was producing a truncated
IRX (-201 from the IOP loader = bad header). `bin2c` sidesteps the entire
class of issues.

## Defines + flags

`Makefile.eeglobal_cpp` prepends its own `-D_EE -G0 -O2` etc., so the
addon's defines must NOT collide:

```make
DEFINES = \
    -DPOLYPHASE_PLATFORM_ADDON=1 \
    -DPLATFORM_PS2=1 \
    -DAPI_PS2GS=1 \
    -DEDITOR=0 \
    -DLUA_ENABLED=1 \
    -DGLM_FORCE_RADIANS \
    -DGLM_FORCE_UNRESTRICTED_GENTYPE \
    -DLOGGING_ENABLED=1 \
    $(addprefix -D,$(ADDON_DEFINES))

EE_CFLAGS   = -Wall -Wno-unused-parameter -Wno-unused-variable $(DEFINES)
EE_CXXFLAGS = $(EE_CFLAGS) -fno-exceptions -fno-rtti -std=gnu++17
```

`-DGLM_FORCE_UNRESTRICTED_GENTYPE` ‚Äî see [Architecture.md](Architecture.md).
Required because the R5900 FPU isn't IEC559-conformant and GLM's
`static_assert(is_iec559 || is_integer)` rejects PS2 `float`.

**Do not add `-G0` to `EE_CFLAGS`.** `Makefile.eeglobal_cpp` already
supplies it, and overriding it via a duplicate `-G0` *can* break the
small-data area which PS2 link relies on. The duplication is also
silently masked because gcc accepts repeated `-G`.

`-fno-exceptions -fno-rtti` ‚Äî PS2 builds without RTTI or exceptions.
Engine code paths that rely on `dynamic_cast<T*>` or `try/catch` need
alternative codepaths gated on `#if PLATFORM_PS2`. (Most existing engine
code already handles this; the engine is targeted at retro consoles
that don't support RTTI either.)

## Link order

PS2 link order is strict left-to-right:

```make
EE_LIBS = \
    -lgskit_toolkit -lgskit -ldmakit \
    -laudsrv -lpad -lmc \
    -lfileXio -lpatches -ldebug \
    -lm -lc -lkernel -latomic \
    $(addprefix -l,$(ADDON_LIBS))
```

Rationale:

- **gsKit chain**: `gskit_toolkit` ‚Üí `gskit` ‚Üí `dmakit` (higher-level
  first, dmaKit at the bottom).
- **PS2 subsystems**: `audsrv` (EE-side stubs that RPC to the IRX),
  `pad`, `mc`, `fileXio`.
- **Patches** (`sbv_patches` ‚Äî module-load restriction lifters), **debug**
  (`scr_printf`).
- **Standard C/C++ runtime**: `m`, `c`, `kernel`, `atomic`.

`-latomic`: engine's `std::atomic<uint64_t>` / `atomic<bool>` calls need
libatomic for sub-word RMW intrinsics (`__atomic_fetch_add_4`,
`__atomic_exchange_1`) that the EE GCC backend doesn't lower to native
MIPS atomic instructions.

`-Wl,--allow-multiple-definition`: the video addon's `StbImageJpeg.cpp`
also defines `STB_IMAGE_IMPLEMENTATION`, colliding with the engine's
`stb_implementation.cpp`. Same workaround the PSP build uses for
newlib/pspkernel `strtol` collisions.

Library files are all-lowercase under `$(PS2DEV)/gsKit/lib/`
(`libgskit.a`, `libdmakit.a`) despite docs and headers using mixed-case
`GSGLOBAL`/`GSKIT_*` types. Be careful when copying examples from gsKit
demos ‚Äî they sometimes capitalise the `-l` name and silently fail to
link.

## Output staging

`Makefile.eeglobal_cpp` links the ELF into `$(CURDIR) = Intermediate/PS2/`.
The Makefile's `stage` rule copies it to `Build/PS2/` so the editor's
`GetCompiledBinaryPath` callback finds it:

```make
.DEFAULT_GOAL := all
all: stage

stage: $(EE_BIN)
	@mkdir -p $(PROJECT_ROOT)/Build/PS2
	@cp $(EE_BIN) $(PROJECT_ROOT)/Build/PS2/$(TARGET).elf
	@echo "Staged $(PROJECT_ROOT)/Build/PS2/$(TARGET).elf"

cleanstage:
	@rm -rf $(PROJECT_ROOT)/Build/PS2
```

`.DEFAULT_GOAL := all` is mandatory: `Makefile.eeglobal_cpp` defines a
rule for `$(EE_BIN)` early in its body, and GNU make picks the FIRST
target encountered in the file as the default. Without the explicit
declaration, the default becomes `$(EE_BIN)` and the .elf builds in
`Intermediate/PS2/` but never copies to `Build/PS2/`.

`include $(PS2SDK)/samples/Makefile.pref` defines toolchain variables
(`EE_CC`, `EE_CXX`, `EE_LD`, `EE_AR`, `EE_TOOL_PREFIX =
mips64r5900el-ps2-elf-`). Without `Makefile.pref`, `EE_CXX` expands to
empty and the compile recipe becomes `-D_EE -G0 ... -c file.cpp -o
file.o`, which the shell parses with the leading `-` stripped, ending up
as `D_EE: command not found` (error 127). Sample Makefiles always
include `Makefile.pref` BEFORE `Makefile.eeglobal[_cpp]`; we do the same.

## ISO output (optional)

When `ps2.makeIso=1` in the Build Profile, the editor invokes `mkisofs`
after the ELF is staged:

```
mkisofs -V POLY00001 -joliet -udf -l -no-emul-boot \
        -boot-load-size 4 -b SYSTEM.CNF \
        -o Build/PS2/<TARGET>.iso Build/PS2/
```

`SYSTEM.CNF` (auto-generated by the addon's `PostPackage`) tells the PS2
BIOS which ELF to load and what disc ID to claim. The resulting ISO
boots in PCSX2 and on real hardware (via Free MCBoot / OPL / disc burn).

## When something fails

- **`D_EE: command not found`** ‚Äî `Makefile.pref` not included. Check
  the include order in the Makefile.
- **Link fails with `multiple definition of stbi_...`** ‚Äî the
  `--allow-multiple-definition` linker flag got stripped or the video
  addon was updated to also include `STB_IMAGE_IMPLEMENTATION`. Either
  is the proximate cause.
- **`PS2SDK not found at .../Makefile.eeglobal_cpp`** ‚Äî ps2dev install
  is incomplete or the sentinel path is wrong. Run `ls
  $(PS2DEV)/ps2sdk/samples/` to verify.
- **Build hangs at `audsrv_irx.c`** ‚Äî `bin2c` is missing or PS2DEV/bin
  isn't on `PATH`. Check `$(PS2SDK)/bin/bin2c --version`.
- **Link fails with `undefined reference to audsrv_*`** ‚Äî `-laudsrv`
  dropped from `EE_LIBS`. The EE-side libaudsrv stubs RPC to the IRX,
  so even though audsrv runs on the IOP, the EE still links against
  the stub library.
- **Forward-reference build break in `Graphics_PS2GS.cpp` or
  `System_PS2.cpp`** ‚Äî see the "Forward references in compilation unit"
  note in [`README.md`](../README.md). Helper declarations must come
  above their first user; the PS2 build is single-translation-unit
  with no LTO.
